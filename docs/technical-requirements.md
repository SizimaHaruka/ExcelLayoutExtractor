# 技術要件書

## 1. 文書の目的

本書は、ExcelLayoutExtractor を実装する際の技術選定、アーキテクチャ方針、非機能要件、開発制約を定義する。

## 2. 技術方針

### 2.1 基本方針

- Excel は入力手段として扱う
- JSON を正本データとする
- 抽出処理とレンダリング処理を分離する
- MVP では精度よりも安定性と追跡性を優先する
- OSS 公開を前提に、依存と責務を明確化する

### 2.2 実装言語

- C#

選定理由:

- Excel 帳票処理ライブラリが充実している
- Windows 環境での利用・検証と相性が良い
- 将来的な GUI 実装との親和性が高い
- 型定義とデータモデルを扱いやすい

### 2.3 代替候補

- Python
- Java

## 3. 想定アーキテクチャ

```text
Excel Template
    ↓
Excel Parser
    ↓
Normalized Sheet Model
    ↓
Layout Extractors
    ↓
Layout JSON
    ↓
Preview Renderer / Editor / Exporter
```

## 5. ライブラリ選定方針

### 5.1 Excel 読み込み候補

- `ClosedXML`
- `EPPlus`
- `DocumentFormat.OpenXml`

### 5.2 推奨

MVP は `ClosedXML` または `EPPlus` を利用し、必要に応じて Open XML レベルの補完を行う。

選定基準:

- セル、行高、列幅、罫線、マージの取得しやすさ
- 印刷設定の取得可否
- 画像や図形の扱いやすさ
- 実装速度
- ライセンス条件

## 6. データモデル要件

### 6.1 ページモデル

- ページサイズ
- 単位
- 余白
- 印刷範囲

### 6.2 要素モデル

- 共通属性: `type`, `x`, `y`
- 種別ごとの拡張属性
- スタイル属性の保持

### 6.3 中間モデル

- Excel 生データと最終 JSON の間に、正規化済みの中間モデルを置く
- セル座標、スタイル、罫線、結合情報を保持する
- 後続抽出器が Excel ライブラリに直接依存しない構造にする

## 7. 非機能要件

### 7.1 保守性

- 各抽出処理は独立モジュール化する
- 入出力境界を明確にする
- ロジックはテスト可能な単位に分離する

### 7.2 追跡性

- 出力要素は元セル範囲を辿れることが望ましい
- 警告とエラーは対象シート、セル、要素種別を含める

### 7.3 性能

- 単票または単純帳票を数秒以内に解析できることを目安とする
- 大規模ブック全体最適化は初期対象外とする

### 7.4 安定性

- 同一入力に対して再現性のある結果を返す
- 未対応要素が存在しても可能な範囲で解析を継続する

## 8. 単位変換要件

### 8.1 必須

- 行高を mm に変換できること
- 列幅を mm に変換できること
- 印刷範囲起点で絶対座標を算出できること

### 8.2 注意点

- Excel 列幅は文字幅基準で定義されるため、単純変換では誤差が出る
- 変換式はライブラリ依存ではなく、検証可能な形で実装する

## 9. 線抽出技術要件

- 罫線を上下左右のエッジとして正規化できること
- 重複線分を除去できること
- 線種、太さ、色の差異を保持できること
- 近接線分のマージ条件を明文化できること

## 10. テキスト・変数抽出技術要件

- セル文字列と表示書式を取得できること
- 変数構文を正規表現または構文解析で検出できること
- 変数キーと書式指定を分離できること
- 不正構文を警告として扱えること

## 11. 画像抽出技術要件

- 埋め込み画像の位置とサイズを取得できること
- JSON 側で参照可能な識別子を付与できること
- MVP では完全な画像種別判定までは求めない

## 12. プレビュー技術要件

- HTML 単体で表示できること
- JSON から直接描画できること
- デバッグしやすいこと
- 後続の UI エディタへ流用しやすいこと

推奨:

- SVG ベース描画
- あるいは絶対配置 HTML

## 13. テスト要件

### 13.1 ユニットテスト対象

- 単位変換
- 変数パース
- 線分マージ
- JSON シリアライズ

### 13.2 結合テスト対象

- サンプル Excel からの抽出結果
- 既知帳票との比較
- 代表帳票 3 種程度による回帰確認

## 14. セキュリティ・安全性

- マクロを実行しない
- 外部リンクを解決しない
- 未知形式は安全側で失敗させる
- OSS 公開を前提に、商用制約の強い依存は避ける

## 15. 今後の拡張を見据えた要件

- table 推定を追加できる構造であること
- 修正 UI を追加できること
- PDF レンダラへ接続できること
- CLI とライブラリ利用の両方に展開できること
